> 自动加载：本文件 | 详细文档按需读取：`docs/`

## 语言

全中文输出：所有对话、文档、注释、commit message 一律使用中文。

## 仓库定位

这个仓库不是业务代码仓，而是一个可迁移的 AI 协作开发框架仓。

核心目标：

- 把 vibe coding 做成可重复执行的流程系统。
- 把多 Agent 协作拆成明确角色和责任边界。
- 把错误、复盘、门禁、文档沉淀成长期资产。
- 把“愿景生长”方法从具体项目里抽出来，复用到新项目。

原始提炼来源：`Greyfield` 项目。

## 生长愿景模式

本框架默认采用“愿景生长”模式，而不是一开始就做重规划、重拆解、重里程碑承诺。

核心做法：

- 长期愿景由 `docs/growth/architecture-v2.md` 和 `docs/growth/context-runtime.md` 锁定。
- 当前实现边界由 `docs/growth/spine-now.md` 冻结。
- 先让最小 spine 跑起来，再逐步长出新能力。
- 不在当前 spine 里的能力，默认不提前做。

## 核心层次

- `docs/workflows/`：定义任务从进入到完成的主流程。
- `docs/personas/`：定义多角色协作方式。
- `docs/rules/`：按任务类型拆分的硬规则。
- `docs/runbooks/error-books/`：定义错题本阅读和落盘方式。
- `docs/growth/`：愿景生长方法和样例文档。
- `docs/templates/`：新项目可复用的模板。

## 任务类型与推荐流程

| 任务类型 | 判断标准 | 推荐流程 |
|----------|----------|----------|
| 文档更新 | 只改说明、规则、模板 | 直接修改 + 自查 |
| 单文件小修 | 影响范围小，逻辑明确 | 直接实施 + 最小验证 |
| Bug 修复 | 有复现或明确异常 | 读错题本 + 修复 + 独立 CR + 回归 |
| 多文件功能 | 跨多个文件或多个模块 | 方案确认 + 实施 + 独立 CR + 验证 |
| 新能力生长 | 超出当前 spine | 愿景对齐 + 架构卡位 + 冻结范围 + Mini SR + 实施 |

拿不准时，按“多文件功能”处理。

## 四个硬门禁

### 1. 任务入口门禁

在开始分析或修改前先说清楚：

- 任务是什么
- 为什么属于这个任务类型
- 对应走哪条流程
- 这次读了哪些错题本
- 工作分支 / worktree 是否就绪

### 2. 方案确认门禁

多文件功能进入实施前，至少要冻结：

- 验收标准
- 技术方案
- 风险点
- 改动文件清单
- 文档落盘位置

### 3. 完成门禁

代码写完不等于完成。进入“完成”前必须明确：

- 自验证是否真的做过
- 独立 Code Review 是否完成
- P0 / P1 是否归零
- P2 是否逐条处置

### 4. 落盘门禁

遇到以下场景要自动写入错题本：

- CR 发现高优问题
- 测试失败
- 同一错误连续出现
- 用户指出流程违规
- 实现与设计明显不一致

## 错题本加载策略

每次任务默认先读：

1. `docs/runbooks/error-books/_index.md`
2. `docs/runbooks/error-books/flow-rules.md`

再按任务类型读取：

- 走流程：`flow-gate.md`
- 改代码：`flow-code-habit.md`
- 写设计：`flow-design.md`
- 改接口：`interface-rules.md`
- 工具踩坑：`tool-rules.md`

发现新错误时，必须补记到对应错题本，并根据需要更新 checklist 或规则。

## 多角色协作

多人或多 Agent 协作时，优先使用这些角色：

- Architect：边界、架构、接口
- Tech Lead：实现路径、风险、拆分
- Developer：编码和落地
- QA Lead：验证、测试、质量风险
- Human Proxy PM：需求约束、用户视角、验收标准
- Recorder：会议纪要、决策落盘、索引维护

不要让单个 Agent 同时扮演所有角色并直接给自己背书。

## 使用原则

- 先让流程跑通，再扩复杂度。
- 先冻结当前边界，再开始扩写。
- 先做真实验证，再写“已完成”。
- 先落盘，再假设团队记住了。
- 框架文档要持续抽象和泛化，避免被单个项目重新绑死。
