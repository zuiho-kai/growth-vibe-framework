# 生长愿景流程 Vibe Code 开发框架

把 AI 开发从“想到就写、写完就算”的 vibe coding，升级成“有愿景锚点、有冻结边界、有门禁闭环、有错误沉淀”的生长式工程流程。

本仓库主要提炼自原项目 [Greyfield](https://github.com/zuiho-kai/Greyfield) 中已经实战跑出来的愿景生长体系、流程体系、角色体系、错题本体系和规则体系。它不是业务代码模板，而是一套可以迁移到新项目里的开发框架。

## 一眼看懂这套流程

```mermaid
flowchart LR
    A["需求 / 想法输入"] --> B["任务入口门禁"]
    B --> C["愿景对齐"]
    C --> D["架构卡位"]
    D --> E["冻结当前 Spine"]
    E --> F["Mini SR / 验收确认"]
    F --> G["开发实施"]
    G --> H["独立 Code Review"]
    H --> I["真实验证 / ST"]
    I --> J["完成门禁"]
    J --> K["错误落盘 / 更新错题本"]
    K --> L["下一轮继续生长"]
```

这套框架的核心不是“让流程变多”，而是把最容易失控的几个点钉死：

- 愿景没对齐，不能直接做新能力
- 当前边界没冻结，不能边写边长
- 用户没确认验收标准，不能默认“我觉得这样就行”
- 代码没经过独立 CR，不能算完成
- 验证没闭环，不能推进到下一阶段
- 错误没落盘，下次大概率还会再犯

## 普通 Vibe Coding vs 生长架构 Flow

| 维度 | 普通 vibe coding | 生长架构 flow |
|------|------------------|---------------|
| 开始方式 | 想到就写 | 先做任务分类，再进门禁 |
| 长期方向 | 靠临时上下文记忆 | 先锁愿景文档，再围绕它生长 |
| 当前范围 | 经常边做边扩 | 先冻结 spine，再实施 |
| 新能力引入 | 容易直接跳做 | 必须先愿景对齐、架构卡位、Mini SR |
| 角色分工 | 一个 Agent 包打一切 | Architect / Tech Lead / QA / Dev / Human Proxy PM 分视角 |
| Code Review | 常常自己审自己 | 强调独立 CR |
| 测试 | 往后补，容易漏 | 验收标准和验证路径前置 |
| 复盘 | 问题过了就算 | 错题本沉淀，回灌 checklist |
| 结果 | 快，但容易飘、容易乱 | 也快，但更稳、更可接力、更能积累 |

## 完整流程

| 阶段 | 名称 | 主要参与者 | 目标 | 产物 |
|------|------|------------|------|------|
| 0 | 需求输入 | 用户 / 主 Agent | 明确问题、范围、目标 | 任务描述、背景、边界 |
| 1 | 任务入口门禁 | 主 Agent | 判断任务类型、读取对应规则和错题本 | 门禁声明、任务路径 |
| 2 | 愿景对齐 | Architect / Human Proxy PM | 确认当前需求是否符合长期愿景 | 一句话定位、与愿景关系 |
| 3 | 架构卡位 | Architect / Tech Lead | 明确模块边界、数据流、依赖关系 | 方案草图、参考实现、改动面 |
| 4 | Spine 冻结 | Architect / Human Proxy PM | 锁定当前轮次必须做与明确不做的范围 | `spine-now` 更新 |
| 5 | Mini SR / 验收确认 | Tech Lead / QA Lead / 用户 | 把“怎么才算做完”说清楚 | 验收 checklist、风险点 |
| 6 | 开发实施 | Developer / 多 Agent | 按冻结范围开发，不临场扩 scope | 代码、文档、进度记录 |
| 7 | 独立 Code Review | 独立 CR Agent | 独立找出 P0/P1/P2 风险 | CR 结果、修复记录 |
| 8 | 真实验证 / ST | QA Lead / Developer | 验证真实链路闭环，不做假验证 | 测试结果、验证记录 |
| 9 | 完成门禁 | 主 Agent / QA Lead | 明确是否真的可交付 | 完成门禁声明 |
| 10 | 错误落盘 | 主 Agent / 相关角色 | 把错误沉淀进错题本和清单 | error-book 更新、checklist 更新 |
| 11 | 下一轮生长 | 多方协作 | 从当前 spine 继续长出下一层能力 | 新一轮 frozen scope |

## 愿景生长到底在长什么

这一步不是“先写一份很大的 roadmap”，而是把项目拆成两层：

- 长期不轻易变的方向：愿景、系统位置、核心能力模型
- 当前轮次真正要做的最小 spine：这次必须活起来的那一小段

也就是说，生长架构不是不要设计，而是把设计分成“长期锚点”和“当前冻结边界”。

| 层次 | 关注点 | 会回答什么问题 | 典型产出 |
|------|--------|----------------|----------|
| 愿景层 | 项目为什么存在、最终想长成什么 | 这件事是否值得长期做？它在整体里处于什么位置？ | `architecture-v2.md`、`context-runtime.md` |
| Spine 层 | 当前这一轮到底做多小、多准 | 这次必须有什么？明确不做什么？ | `spine-now.md` |
| 实施层 | 怎么落成代码和可验证产物 | 需要改哪些文件？怎样验证？哪里有风险？ | Mini SR、实现、测试记录 |
| 复盘层 | 这次哪里出错、下次怎么拦 | 是流程问题、工具问题还是架构问题？ | 错题本、checklist 更新 |

### 生长架构后的标准产出

| 产出 | 说明 |
|------|------|
| 愿景定位 | 说明新需求和长期方向的关系，不让项目乱长 |
| 边界冻结 | 明确“这次做什么 / 不做什么”，防止 scope 漂移 |
| 验收标准 | 防止所有人心里各自有一个 done |
| 风险清单 | 先把技术风险、验证风险、体验风险摊开 |
| 错误沉淀 | 每次出错都反向增强流程，而不是只修当前问题 |

## 每个阶段为什么必须有

| 阶段 | 为什么必须有 | 如果跳过，常见后果是什么 |
|------|--------------|--------------------------|
| 任务入口门禁 | 防止一上来就动手、流程失配 | 明明该走设计流程，却直接开写 |
| 愿景对齐 | 防止需求虽然能做，但把项目做歪 | 短期看有产出，长期越来越乱 |
| 架构卡位 | 提前把边界、接口、依赖说清楚 | 做着做着发现模块耦合爆炸 |
| Spine 冻结 | 锁住当前范围，减少中途飘移 | 越写越多，永远做不完 |
| Mini SR | 把“做完标准”前置 | 做完后才发现大家理解的 done 不一样 |
| 开发实施 | 严格按冻结边界执行 | 规格和实现慢慢分叉 |
| 独立 CR | 避免自己审自己 | 很多问题会被“合理化”掉 |
| 真实验证 / ST | 验证真实链路，不接受假绿 | 语法都对，实际一跑就断 |
| 完成门禁 | 防止“差不多了”提前收工 | CR 没收口、验证没闭环就算完成 |
| 错误落盘 | 让错误变成资产 | 同类错误不断重复，没有学习效应 |

## 这套流程的卖点

### 生长而不是失控扩张

很多项目的问题不是没有想法，而是所有想法都在同一层直接落地，最后项目会越来越重、越来越乱。

这套框架的核心做法是：

- 先锁长期愿景
- 再冻结当前 spine
- 只让当前轮次长出最小可用能力

这样做的直接价值是：

- 长期方向不会丢
- 当前实现不会飘
- 每轮都能明确交付

### 强门禁，不靠自觉

很多团队不是没有流程，而是流程只是“建议”。

这套框架把关键节点做成硬门禁：

- 任务入口门禁
- 方案确认 / Mini SR 门禁
- 修复门禁
- 完成门禁
- 错误落盘门禁

它的直接价值是减少这些常见失控场景：

- 还没分析完就直接写代码
- 当前范围没锁定就越做越大
- 只测几个 happy path 就宣布 done
- CR 没收口就推进到下一阶段
- 一个坑来回踩很多次

### 多角色制衡，降低单 Agent 幻觉

普通 vibe coding 的常见问题，是一个 Agent 同时扮演 PM、架构师、开发、测试和 reviewer，最后一路自洽，一路跑偏。

这套框架通过角色拆分，把不同视角强制拉进来，减少这种幻觉。

| 角色 | 关注点 | 会提出什么问题 | 典型产出 |
|------|--------|----------------|----------|
| Architect | 架构边界、模块关系、长期演进 | 这件事放进系统后会不会把结构做脏？ | 架构约束、模块边界、技术红线 |
| Tech Lead | 技术可行性、复杂度、依赖顺序 | 真做下来难点在哪？先做什么后做什么？ | 技术方案、改动面、依赖关系 |
| QA Lead | 可测试性、边界条件、失败路径 | 这个方案怎么测？哪里最容易漏？ | 测试风险、验收条件、验证路径 |
| Developer | 实施成本、联动风险、具体落地 | 需要改哪些文件？最可能在哪出 bug？ | 落地路径、实现清单 |
| Human Proxy PM | 用户价值、范围控制、优先级 | 这是用户真正要的吗？最小交付是什么？ | 范围约束、优先级建议 |
| Recorder | 决策沉淀、纪要整理、索引维护 | 这次讨论和教训有没有被落盘？ | 讨论纪要、进度更新、索引同步 |

### 错题本驱动，越做越稳

`docs/runbooks/error-books/` 是这套体系的核心资产之一。

这里记录的是实战中真实发生过的错误，包括：

- 流程错误
- 工具使用错误
- 前后端接口错误
- React / UI 错误
- 架构与边界控制错误

这意味着：

- 新任务开始前可以预读
- 出错后可以快速归因
- 做完后还能反向更新 checklist
- 项目经验不会只存在于聊天记录里

### 独立评审，把“差不多”变成“真的过了”

这套流程强调：

- 设计确认和代码评审分开
- 编码 Agent 和 CR Agent 分开
- 实现完成和验证完成分开
- 修 bug 和归因落盘分开

核心目的是把“我觉得没问题”换成“独立检查确认过没问题”。

## 仓库结构

| 路径 | 作用 |
|------|------|
| `CLAUDE.md` | 全局协作规则、门禁、流程总控 |
| `docs/workflows/` | 完整流程、清单、协作规范 |
| `docs/personas/` | 多角色定义 |
| `docs/rules/` | 按任务类型拆分的硬规则 |
| `docs/runbooks/error-books/` | 错题本、流程教训、常见失败模式 |
| `docs/runbooks/` | 模型选择、团队管理、进度分层等手册 |
| `docs/growth/` | 愿景生长样例，包括愿景、spine、索引方式 |
| `docs/templates/` | 文档模板 |
| `docs/specs/` | 历史规格样例，可按项目替换 |
| `docs/tests/` | 测试文档样例 |
| `claude-progress.txt` | 关键决策和阶段进度记录 |

## 建议怎么开始

1. 先读 `CLAUDE.md`
2. 再读 `docs/growth/README.md`
3. 看 `docs/workflows/` 理解任务路径
4. 看 `docs/personas/roles.md` 理解角色分工
5. 开工前读 `docs/runbooks/error-books/_index.md`
6. 用 `docs/growth/` 里的愿景文档骨架替换成你自己的项目内容

## 适合谁

- 想把 AI coding 从“随缘发挥”变成“有纪律开发”的个人
- 想做多 Agent 协作研发的团队
- 想把 bug、复盘、规则沉淀成长期资产的项目
- 想用愿景生长方式推进高不确定度产品的人
- 想搭自己的 Claude Code / Codex / Agent 工作流底座的人

本仓库当前内容主要提炼自 [Greyfield](https://github.com/zuiho-kai/Greyfield)。
