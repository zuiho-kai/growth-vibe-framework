# PID Vibe Code Framework

把 AI 开发从“随缘写码”升级成“有流程、有分工、有门禁、有复盘”的一套完整框架。

本仓库主要提炼自原项目 [bot_civ](https://github.com/zuiho-kai/bot_civ) 中已经实战验证过的流程体系、角色体系、错题本体系和文档体系，目标不是提供业务代码，而是提供一套可直接复用的 PID 式 vibe code 开发底座。

## 这套框架到底是什么

它不是一个空白模板仓库，也不是单纯的 prompt 集合。

它更像一套把 AI 协作开发“工程化”的方法论，核心思路是：

- 需求不能直接变代码，要先过讨论和设计
- 设计不能直接默认正确，要过多角色评审
- 实现不能自己说完成，要过独立 CR 和测试
- 错误不能修完就算，要沉淀进错题本
- 讨论不能停留在聊天记录，要变成规格、纪要和决策记录

## 完整流程

这套流程不是一句“先想清楚再开工”，而是明确拆成下面这些阶段。

| 阶段 | 名称 | 主要参与者 | 核心目标 | 主要产物 |
|------|------|------------|----------|----------|
| 0 | 需求输入 | 用户 / 主 Agent | 明确要解决的问题 | 任务描述、背景、边界 |
| 1 | 五方脑暴 | Architect / Tech Lead / QA Lead / Developer / Human Proxy PM | 先把方向聊透，不急着写方案 | 脑暴纪要、候选方案、关键分歧、待拍板决策 |
| 2 | 用户拍板 | 用户 | 对关键方向做明确选择 | 决策结果 |
| 3 | IR 草稿 | 设计终端 | 把方向写成需求原型 | IR 文档 |
| 4 | 五方评审 IR | 五方角色 | 检查 IR 是否漏项、跑偏、不可验收 | IR 评审纪要、修订后的 IR |
| 5 | SR 系统需求 | 设计终端 | 把 IR 拆成可执行的系统级要求 | SR 文档 |
| 6 | AR 架构设计 | 设计终端 | 把异常、边界、安全、时序、风险补齐 | AR 文档 |
| 7 | UI 设计稿 | 设计终端 | 前端场景下先锁定视觉与交互验收标准 | UI 设计稿 |
| 8 | 正向串讲 | Developer -> QA Lead | 开发向 QA 讲清楚设计意图 | 串讲记录、补充测试点 |
| 9 | 反向串讲 | QA Lead -> Developer | QA 用自己的理解复述，确认一致 | 测试场景列表、补充修订 |
| 10 | 测试设计 | QA Lead | 在写代码前设计测试用例 | TEST 文档 |
| 11 | 开发实施 | Developer / 多 Agent | 按 SR/AR 实现，不自由发挥 | 代码、单测、进度记录 |
| 12 | 独立 Code Review | 独立 CR Agent | 不是自己审自己，清掉 P0/P1/P2 风险 | CR 结果、修复记录 |
| 13 | UI 审美验收 | 独立审查 Agent | 前端场景下独立检查视觉和交互质量 | UI review 文档 |
| 14 | ST / 集成测试 | QA Lead / Developer | 验证真实链路是否闭环 | 测试报告 |
| 15 | 上线总结 | 多方确认 | 确认可交付并更新进度 | RELEASE / progress 更新 |
| 16 | 错误落盘 | 主 Agent / 相关角色 | 把这次踩坑反写进错题本 | error-book 更新、checklist 更新 |

## 五方脑暴到底脑暴什么

这一步是整套流程最关键的入口。它不是“大家随便聊聊”，而是把同一个需求拆成五种视角，避免一开始就被单一思路带偏。

| 角色 | 关注点 | 会提出什么问题 | 典型产出 |
|------|--------|----------------|----------|
| Architect | 架构边界、模块关系、未来演进 | 这件事是否符合现有架构？会不会把系统做脏？ | 架构约束、模块拆分建议、技术红线 |
| Tech Lead | 技术可行性、实现复杂度、依赖关系 | 真做下来难点在哪？先做什么后做什么？ | 技术方案、依赖顺序、工作量判断 |
| QA Lead | 可测试性、边界条件、失败路径 | 这个方案怎么测？哪里最容易漏测？ | 测试风险、异常场景、验收条件 |
| Developer | 实施成本、改动范围、实现细节 | 需要改哪些文件？是否会引入联动风险？ | 落地路径、代码改动面、实现疑问 |
| Human Proxy PM | 用户价值、范围控制、优先级 | 这是不是用户真正要的？有没有更小可交付路径？ | 范围约束、优先级建议、拍板议题 |

### 五方脑暴后的标准产出

| 产出 | 说明 |
|------|------|
| 候选方案列表 | 不只一个方案，至少比较主方案和备选方案 |
| 关键分歧点 | 哪些地方团队内部有不同判断 |
| 风险清单 | 架构风险、交互风险、测试风险、实现风险 |
| 用户待拍板问题 | 哪些问题必须由用户决定，不能 AI 自己替用户拍板 |
| 脑暴纪要 | 保留讨论过程，不让上下文在会话结束后消失 |

## 每个阶段的细节解释

| 阶段 | 为什么必须有 | 如果跳过，常见后果是什么 |
|------|--------------|--------------------------|
| 需求输入 | 明确任务是什么，不让后面所有讨论建立在误解上 | 做了很多事，但不是用户真正要的 |
| 五方脑暴 | 先把方向、风险、边界摊开 | 一上来就开写，后面返工巨大 |
| 用户拍板 | AI 不能替用户做关键业务决策 | 团队默认了一个方向，最后用户不买单 |
| IR | 把“想法”变成“可以讨论的需求原型” | 讨论停留在口头，没有稳定输入 |
| 五方评审 IR | 检查需求原型有没有漏项、伪需求、不可验收项 | 方向看似清楚，落地时发现洞很多 |
| SR | 把需求拆成系统可执行项 | 开发只能靠猜，不知道具体改什么 |
| AR | 补齐异常、时序、安全、分支、风险 | happy path 看起来通，但一到边界就炸 |
| UI 设计稿 | 前端功能先锁视觉和交互标准 | 做完能用，但丑、乱、不统一 |
| 正向串讲 | 开发把自己的理解说出来 | QA 不知道开发实际上想做什么 |
| 反向串讲 | QA 用自己的话复述，查理解偏差 | 双方都以为对方懂了，实际没对齐 |
| 测试设计 | 先想好怎么验证，再进入实现 | 代码写完后才临时补测，漏测严重 |
| 开发实施 | 按设计执行，而不是临场发挥 | 规格和实现慢慢分叉 |
| 独立 CR | 必须由独立视角找问题 | 自己审自己，容易把问题合理化 |
| UI 审美验收 | 前端不仅要能用，还要达到视觉标准 | 界面功能没错，但体验差 |
| ST / 集成测试 | 验证真实链路闭环，不只看单点通过 | 单测都绿，系统一跑就断 |
| 上线总结 | 做完要留下可交付痕迹和阶段状态 | 下次回头看，不知道做到哪了 |
| 错误落盘 | 让错误变成资产，而不是一次性损耗 | 同类错误反复出现，团队没有学习效应 |

## 这套流程的卖点

### 1. 强门禁，不靠自觉

很多流程之所以失效，不是因为团队不知道流程，而是因为流程只是“建议”。

这套框架把关键节点做成硬门禁：

- 任务入口门禁
- 里程碑门禁
- Phase 完成门禁
- 错误落盘门禁

它的优势是能明显降低以下问题：

- 还没分析完就直接写代码
- 还没评审就默认方案成立
- 测试没跑完整就宣布 done
- CR 没收口就推进下阶段
- 一个坑反复踩十次

### 2. 多角色制衡，避免单 Agent 幻觉

普通 vibe coding 最大的问题之一，是一个 Agent 同时在扮演 PM、架构师、开发、测试和 reviewer，最后非常容易“一路自洽，一路跑偏”。

这套框架通过角色拆分，强制引入不同视角，降低这种幻觉。

### 3. 错题本驱动，越做越稳

`docs/runbooks/error-books/` 是整套体系里很有价值的一层。

这里记录的不是抽象原则，而是实战里真实发生过的错误：

- 流程错误
- 工具使用错误
- 前后端接口错误
- React/UI 错误
- Agent/后端/数据库错误

这意味着新任务开始前可以预读，出错后可以快速归因，做完后还能反向更新 checklist。

### 4. 独立评审，减少“差不多就行”

这套流程强调：

- 设计评审和代码评审分开
- 编码和审查分开
- 前端实现和 UI 验收分开

核心目的是把“我觉得没问题”变成“独立检查确认过没问题”。

### 5. 适合复杂任务，不只适合 demo

它最适合以下场景：

- 多文件改动
- 前后端联动
- 复杂新功能
- 需要多轮讨论和评审的需求
- 希望把 AI 开发沉淀成长期工作流的团队

## 仓库结构

| 路径 | 作用 |
|------|------|
| `CLAUDE.md` | 全局协作规则、门禁、流程总控 |
| `docs/workflows/` | 完整流程、清单、协作规范 |
| `docs/personas/` | 五方角色和其他角色定义 |
| `docs/runbooks/error-books/` | 错题本、流程教训、常见失败模式 |
| `docs/runbooks/` | 模型选择、团队管理、进度分层等手册 |
| `docs/templates/` | 文档模板 |
| `docs/specs/` | 规格样例，可按自己项目替换 |
| `docs/tests/` | 测试文档样例 |
| `claude-progress.txt` | 关键决策和阶段进度记录 |

## 建议怎么开始

1. 先读 `CLAUDE.md`
2. 再读 `docs/workflows/development-workflow.md`
3. 看 `docs/personas/roles.md` 理解角色分工
4. 开工前先读 `docs/runbooks/error-books/_index.md`
5. 把你自己的 PRD、spec、progress 替换进去，保留流程骨架

## 来源

本仓库当前内容主要提炼自 [bot_civ](https://github.com/zuiho-kai/bot_civ)。

可以把它理解成：

- `bot_civ` 是具体业务项目
- `pid-vibe-framework` 是把其中已经跑通、踩坑、复盘过的流程系统独立抽出来，变成更通用的开发框架
