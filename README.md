# PID Vibe Code Framework

一套从实战项目里抽出来的 AI 协作开发框架，重点不是“代码模板”，而是“把 vibe coding 做成可复用、可复盘、可控的流程系统”。

本仓库主要抽取自原项目 [bot_civ](https://github.com/zuiho-kai/bot_civ) 的流程层、人设层、错题本层、规格层与协作约束层，用来作为新的 PID 流程开发基座。

## 这套框架卖什么

- 不是空白模板，而是从真实项目里打磨出来的实战框架
- 不是只管写代码，而是覆盖“需求讨论 -> 设计 -> 开发 -> Code Review -> 测试 -> 验收 -> 复盘”的完整闭环
- 不是靠人自觉执行流程，而是通过门禁、清单、角色分工把流程硬化下来
- 不是 bug 修完就结束，而是把错误沉淀进错题本，持续反哺后续开发
- 不是单 Agent 硬冲，而是让 Architect、Tech Lead、QA、Developer、Human Proxy PM 各自承担不同责任

## 这套流程的特点

### 1. 全链路闭环

从需求进入开始，到 IR、SR、AR、开发实施、独立 CR、ST 验证、上线总结，整条链路都有明确产物，不会只停留在“聊过了”和“差不多做完了”。

### 2. 强门禁，防跳步

这套框架最核心的特点，是把很多容易被 vibe coding 跳过的步骤做成硬门禁：

- 任务入口门禁
- 里程碑门禁
- Phase 完成门禁
- 错误落盘门禁

优点是能明显减少这些常见失控场景：

- 还没想清楚就开写
- 设计没锁定就直接改代码
- 测了几个 happy path 就宣布完成
- CR 没做完就推进下一阶段
- 同一个错误反复重犯

### 3. 多角色协作，不让一个 Agent 同时当 PM、架构师、开发和 QA

仓库内置了完整的人设体系，核心角色包括：

- Architect
- Tech Lead
- Developer
- QA Lead
- Human Proxy PM
- Recorder

它的价值不是“角色扮演有趣”，而是强制把视角拆开，让方案、风险、实现、验证分别有人负责，降低单一 Agent 一路自洽到底的幻觉。

### 4. 错题本驱动，而不是经验丢失

`docs/runbooks/error-books/` 是这套框架的另一个核心卖点。

很多团队的问题不是不会写，而是同一类错误会反复发生。这套体系把流程错误、工具错误、前后端接口错误、React/UI 错误、后端/数据库错误都做成可检索、可追加、可回灌的错题本。

它的实际优势是：

- 新任务开始前可按类型预读
- 出问题后能迅速归因
- 流程会随着踩坑记录越来越稳
- 团队知识不会随着对话结束一起消失

### 5. 独立 CR 和独立验收

这套流程强调“不能自己审自己”：

- 设计评审和代码评审分开
- 编码 Agent 和 CR Agent 分开
- 前端 UI 还可以走独立审美验收

优点是把“我觉得差不多”换成“有独立检查者确认过”，对 vibe coding 场景特别重要。

### 6. 适合多 Agent、多终端、长链路任务

仓库里已经包含多终端协作、Team vs Task 分工、模型选择、进度分层、讨论纪要等机制。它比较适合：

- 复杂新功能
- 多文件改动
- 前后端协作
- 需要反复评审的方案设计
- 想把 AI 开发从一次性 prompt 升级成持续工程流程的团队

## 这套流程的实际优点

如果说普通 vibe coding 的特点是“快”，这套 PID 流程框架追求的是“快而不乱”。

它带来的直接收益通常是：

- 返工更少，因为前期讨论和设计更扎实
- 失控更少，因为每个阶段有门禁
- 漏测更少，因为 QA 和测试设计被显式纳入流程
- 幻觉更少，因为多角色会互相制衡
- 知识沉淀更多，因为错误、讨论、设计、决策都会落文档
- 更适合接力，因为后来的人能从文档和错题本快速接上

## 仓库里有什么

- `CLAUDE.md`
  这套框架的总控文件，包含协作规则、门禁、阶段要求、提交流程
- `docs/workflows/`
  完整流程定义，包括开发流程、集成流程、辩论流程、门禁清单
- `docs/personas/`
  多角色人设与协作规则
- `docs/runbooks/error-books/`
  错题本、流程教训、常见失败模式
- `docs/runbooks/`
  模型选择、团队管理、进度分层等操作手册
- `docs/templates/`
  文档模板
- `docs/specs/`
  原项目沉淀下来的规格样例，可直接参考
- `claude-progress.txt`
  进度与关键决策记录

## 建议怎么用

1. 先读 `CLAUDE.md`
2. 再读 `docs/workflows/development-workflow.md`
3. 根据任务类型决定是否启用多角色协作
4. 开工前先读 `docs/runbooks/error-books/_index.md`
5. 把你自己的 PRD、spec、progress 替换进来，保留流程内核

## 适合谁

- 想把 AI coding 从“随机发挥”变成“有纪律开发”的个人
- 想做多 Agent 协作研发的团队
- 想把 bug、复盘、评审变成长期资产的项目
- 想搭自己的 Claude Code / Codex / Agent 工作流底座的人

## 来源

本仓库当前内容主要提炼自原项目 [bot_civ](https://github.com/zuiho-kai/bot_civ)。
