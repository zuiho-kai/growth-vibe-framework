# 生长愿景流程 Vibe Code 开发框架

把 vibe coding 从“一次性 prompt 运气学”，升级成可复制、可接力、可复盘的工程流程系统。

这个仓库抽取自原项目 [Greyfield](https://github.com/zuiho-kai/Greyfield) 的流程层、规则层、错题本、人设协作和愿景生长文档结构。目标不是提供某个业务项目的代码模板，而是沉淀一套能迁移到新项目里的 AI 协作开发底座。

## 这套框架的特点

- 愿景生长，而不是一次性大设计：先锁长期愿景，再冻结当前 spine，让项目按“先活起来、再长出来”的方式推进。
- 强门禁，而不是靠自觉：任务入口、方案确认、修复、完成、落盘都有硬卡点，减少跳步、漏审、假完成。
- 多角色协作，而不是单 Agent 包打天下：Architect、Tech Lead、Developer、QA Lead、Human Proxy PM、Recorder 各自负责不同视角。
- 错题本驱动，而不是经验随对话蒸发：流程错误、工具坑、接口坑、设计坑会落进错题本，后续任务可复读、可复用、可迭代。
- 独立 CR 和验证，而不是“我自己看着没问题”：编码、审查、验收分离，降低自证正确带来的幻觉风险。
- 适合长链路任务，而不是只适合小 demo：尤其适合需求仍在生长、要多轮讨论、多人接力、多 Agent 协作的项目。

## 这套流程的卖点

- 快而不乱：保留 vibe coding 的速度，但把失控点前移拦住。
- 返工更少：先做愿景对齐、边界冻结和风险确认，后面少走弯路。
- 可接力：决策、规则、错题、进度都在文件里，不依赖某段临时对话。
- 可复盘：每次踩坑都能沉淀成下次的防线，而不是重复付学费。
- 可迁移：这套方法可以从 Greyfield 抽出来，继续用在别的产品、别的团队、别的 Agent 组合上。

## 仓库里有什么

- `CLAUDE.md`：框架总控文件，定义协作方式、门禁和加载策略。
- `docs/workflows/`：开发、集成、讨论、worktree 协作等流程文档。
- `docs/personas/`：多角色人设和协作职责。
- `docs/rules/`：按任务类型拆分的硬规则。
- `docs/runbooks/error-books/`：错题本、流程教训和落盘规则。
- `docs/growth/`：从 Greyfield 抽出来的愿景生长文档样例。
- `docs/templates/`：可以直接复用的文档模板。

仓库里也保留了一部分历史规格、讨论和测试文档，作为“流程怎样在真实项目里落地”的参考样本。

## 为什么这个流程适合现在的 AI 开发

普通 vibe coding 最大的问题不是写得慢，而是容易在“还没想清楚就开写”“写完就算完成”“同一种错反复犯”这三个地方失控。

这套框架的优势就在于：

- 在编码前就要求明确任务类型、边界和退出路径。
- 在多文件功能前强制方案确认，防止边写边改、越改越散。
- 在完成后强制独立 CR 和验证，防止“自己觉得差不多”。
- 在出错后强制落盘，防止团队记忆只存在于聊天窗口。

如果说普通 vibe coding 的卖点是“快”，那这套框架追求的是“快而不乱，快而能接力，快而能沉淀”。

## 建议使用方式

1. 先读 `CLAUDE.md`，理解门禁和文档入口。
2. 按任务类型进入 `docs/workflows/` 和 `docs/rules/`。
3. 需要多角色协作时读取 `docs/personas/`。
4. 开工前先读 `docs/runbooks/error-books/_index.md` 和相关子文件。
5. 用 `docs/growth/` 里的愿景生长文档作为你自己项目的参考骨架。
